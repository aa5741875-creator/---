# Паспорт проєкту: Прогнозування товарообігу мережі Amazon

### 1. Назва проекту / задачі
«Прогнозування щоквартального товарообігу міжнародного маркетплейсу Amazon на 18 місяців наперед».

### 2. Тип моделі
Для розробки використовуються моделі аналізу часових рядів (Time Series Forecasting): статистична модель **SARIMA** (базовий baseline) та просунутий алгоритм **Facebook Prophet** для інтеграції екзогенних факторів (сезонних розпродажів).

### 3. Посилання на таски в Jira / Trello / іншу систему
Усі задачі проєкту зафіксовані та відстежуються на інтерактивній KANBAN-дошці GitHub Projects:
* [Посилання на GitHub Projects дошку з актуальними статусами задач](https://github.com)

### 4. Дата запуска проекту / задачі в продакшн
Планова дата інтеграції алгоритму у звітні процеси — кінець поточного навчального семестру (червень 2026 року).

### 5. Розміщення проекту / задачі
* **Репозиторій:** Код проєкту, ТЗ та датасети розміщено у публічному репозиторії GitHub: [https://github.com](https://github.com)
* **Джерела вхідних даних:** Історичні квартальні звіти Amazon про виручку (Revenue) за останні роки, вивантажені з фінансового агрегатора Macrotrends. Дані збережено у файлі `Новая таблица - Лист2.csv`.
* **Результати:** Результати прогнозування виводяться у форматі таблиці (прогнозні значення на 6 кварталів наперед) та інтерактивного графіка (факт/прогноз). Зберігаються у фінальному аналітичному звіті (Markdown/PDF).
* **Передача даних у інші системи:** Прогнозні значення передаються у фінансовий департамент компанії Amazon для середньострокового планування бюджету.

### 6. Регулярність запуску
Процес виконується **раз на квартал** (після офіційної публікації нових фінансових звітів компанії Amazon для інвесторів).

### 7. Замовник
Фінансовий департамент (Finance & Budgeting Department) та Департамент стратегічного планування Amazon.

### 8. Розробник(и)
Data Scientist / Бізнес-аналітик (студент курсу «Бізнес-прогнозування»).

### 9. Яка проблема вирішується
Необхідність точного середньострокового прогнозування виручки для оптимізації витрат на логістику (розподільчі центри, склади) та уникнення касових розривів. Помилки у плануванні призводять до дефіциту ліквідності або перевитрат на утримання складів.

### 10. Бізнес-вимоги до проекту
* Горизонт прогнозування: 18 місяців (6 кварталів наперед).
* Формат виходу: Графік факт/прогноз із відображенням 95% довірчого інтервалу (коридору ризику).
* Прогноз має легко імпортуватись у BI-систему (наприклад, Tableau або Power BI) для візуалізації топ-менеджменту.

### 11. Метрики якості
Успішність проєкту оцінюється за трьома математичними метриками на тестовій вибірці:
* **MAPE** (Mean Absolute Percentage Error) — цільове значення **< 5%**;
* **MAE** (Mean Absolute Error) — оцінка середньої помилки у мільйонах доларів;
* **RMSE** (Root Mean Squared Error) — контроль великих аномальних відхилень.

### 12. Виключення з прогнозу / обмеження
Не прогнозуються точкові продажі окремих категорій товарів або конкретних продавців (Third-Party Sellers). Модель працює суто із загальним (агрегованим) товарообігом усієї мережі Amazon.

### 13. Особливості / обмеження
* **Бізнес-обмеження:** Наявність сильної сезонності. Четвертий квартал (Q4) кожного року показує аномальний стрибок виручки через різдвяні свята та Black Friday. Модель зобов'язана це враховувати.
* **Технічні обмеження:** Обмежена кількість історичних точок (дані публікуються лише раз на 3 місяці).

### 14. Економічна ефективність
* Оптимізація витрат на заморожування капіталу на складах на 10–15% за рахунок точнішого прогнозу попиту.
* Автоматизація процесу прогнозування та відмова від використання сторонніх платних економетричних сервісів.

### 16. Детальний опис проекту
* **Functional requirements:** Система повинна завантажувати CSV-файл з історією, автоматично перевіряти ряд на стаціонарність, навчати модель SARIMA/Prophet, генерувати вектор прогнозів на 6 кроків уперед та будувати фінальний графік.
* **Flow роботи:** 
  1. Завантаження `Новая таблица - Лист2.csv` ➔ 
  2. Передобробка даних (форматування дат) ➔ 
  3. Розбиття на Train/Test ➔ 
  4. Навчання моделей ➔ 
  5. Розрахунок метрик якості ➔ 
  6. Генерація прогнозу на 18 місяців.
* **Блок-схема:** Візуалізований процес розробки відображено на Канбан-дошці проєкту.

### 17. Вимоги до даних
* **Структура датасету:** Таблиця CSV із двома полями:
  * `Quarter` (тип: Date, формат YYYY-MM-DD) — дата закінчення звітного кварталу;
  * `Revenue_Millions` (тип: Integer) — чиста виручка Amazon у мільйонах доларів США.
* **Оновлення:** Дані оновлюються раз на 3 місяці. Інтеграція відбувається через ручне або автоматичне завантаження оновленого фінансового файлу.

### 20. Примітки (для ведення проекту)
Під час налаштування моделі SARIMA особливу увагу приділити параметру сезонного лагу `s=4` (оскільки дані є квартальними, повний цикл сезонності повторюється кожні 4 точки).
